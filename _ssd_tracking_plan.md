# MonitorIsland — SSD Write/Wear Tracking: Implementation Plan

_Synthesized by an Opus-Max architect from 14 Sonnet ideation agents + 2 Opus-Max planners._

**Plan:** SSD write + perceived-damage tracking with Claude Code / Codex attribution (2 sudoless sources, NVMe SMART cut)

Add three things to Monitor Island, all sudoless and Apple-Silicon-native, slotting into the existing samplers -> Snapshot -> Smoother -> IslandView pipeline: (1) REAL-TIME system-wide SSD write/read throughput; (2) a CUMULATIVE-OVER-TIME host-bytes-written total that survives reboots, plus a DERIVED best-effort 'perceived damage %' (lifetime host bytes / rated TBW), always flagged like the existing tempBestEffort/aneEstimateNote; (3) PER-PROCESS write attribution naming Claude Code and Codex as first-class culprits, plus a coalesced append-only damage LOG that itself cannot become a wear offender.

TWO data sources, both verified LIVE on this machine (uid 501, no sudo, no entitlement, no CIOReport shim):
- SYSTEM-WIDE: public IOKit IOServiceMatching("IOBlockStorageDriver") -> 'Statistics' dict -> 'Bytes (Write)' / 'Bytes (Read)' (UInt64, reset on reboot). Verified: 6 driver entries, exactly ONE non-zero writer (the internal SSD) at Bytes(Write)=570985172992 (~531 GB); all disk-image/virtual entries read 0. New Disk.swift mirrors Network.swift line-for-line and reuses GPU.swift's exact IOKit iterator+defer idiom.
- PER-PROCESS: ri_diskio_byteswritten in the rusage_info_v4 struct that Workloads.swift residentMB(for:) ALREADY fetches per PID (verified in SDK: field at line 308, struct at line 289, RUSAGE_INFO_V4=4). Zero extra syscalls. ri_proc_start_abstime (line 299, same struct) defeats PID reuse. (ri_logical_writes is NOT in v4 — confirms ri_diskio_byteswritten is both the correct wear field and the only one available without changing struct version.)

NVMe SMART (firmware PERCENTAGE_USED / DATA_UNITS_WRITTEN ground truth) is CUT from v1: the ideation agents directly CONFLICT on whether it is sudoless on Apple Silicon (some get SMARTReadData working via the NVMeSMARTLib vtable; others get kIOReturnNotPermitted / firmware 0x10000003 on the ANS3 controller), it needs a PAC- and vtable-slot-fragile private CFPlugin call for a number that updates ~daily and reads 0% on near-new drives, and it is unnecessary to deliver all three required capabilities. The derived best-effort estimate is the honest, robust primary; a one-line TODO marks where a future verified SMART value could override it and clear the flag.

Persistence is anti-self-wear by construction: ONE append-only JSON-lines file in ~/Library/Application Support/MonitorIsland/disk_damage.jsonl, in-memory accumulation, flushed at most once every 5 min (hard precondition flushInterval>=60) and on clean quit; the cross-reboot lifetime baseline is recovered by tailing the file's last line at startup. ~150 bytes/line x ~288 lines/day ~= 43 KB/day ~= 16 MB/year ~= 0.0000026%/yr of an assumed 600 TBW. Never SQLite/WAL (that is the exact Codex offender pattern). Total: 1 new file (Disk.swift) + 6 edited files, ~250 lines.


## Chosen solutions (simplest 1-2)

### Disk.swift — system-wide write/read rate + persisted lifetime + derived best-effort damage % (DiskSampler + SSDWear + DamageLogger)
- **User sees:** Expanded card gains, after the Net row: a 'Disk' rate row ('↑ 12 MB/s  ↓ 3 MB/s' via the existing fmtRate); a 'WRITE ACTIVITY' sparkline (the existing Sparkline component fed by a second history ring) so write bursts are visible at a glance; and an 'SSD~' row showing cumulative host GB written and a '~X.XX% est' damage estimate. The '~' caption and 'est' suffix are the honesty markers, exactly like the tempBestEffort convention. The row recolors accent->amber->red via the EXISTING Theme.pressureWarn/pressureCritical palette as the estimate crosses 10%/50% (no new colors). Compact pill is untouched (its 4 slots are full by documented design). --dump prints all new fields.
- **Data source:** public IOKit IOServiceMatching("IOBlockStorageDriver") -> ['Statistics']['Bytes (Write)'/'Bytes (Read)'] as UInt64 (resets on reboot) for rate + this-run session bytes; lifetime = persistedBaseline (read from JSONL tail at startup) + thisRunSessionBytes; capacity read once sudoless from the AppleANS controller 'Controller Characteristics' -> 'capacity' (verified 1024000000000) to pick a TBW tier; damage% = lifetimeBytes / (ratedTBW*1e12) * 100, always paired with diskWearBestEffort=true.
- **Why simplest:** Network.swift is a direct line-for-line template (prevX/prevTime/primed, elapsed=max(0.001,...), Double(cur &- prev)/elapsed, max(0,...) clamp) and GPU.swift gives the exact IOKit enumeration idiom (IOServiceMatching + IOServiceGetMatchingServices(kIOMainPortDefault,...) + IORegistryEntryCreateCFProperties + defer{IOObjectRelease}). All public IOKit + Foundation: zero new linker flags, zero CIOReport shim. Verified live that summing non-zero entries == the single internal-SSD total (~531 GB), so no IORegistry parent-walking is needed.

### Per-process culprit attribution via ri_diskio_byteswritten (extend the existing Workloads.swift rusage call, accumulate per label)
- **User sees:** Under each Claude Code / Codex workload group in the expanded card, a faint secondary line 'wrote 312 MB' (mono 11pt, Theme.textFaint) showing block bytes written by that tool since Monitor Island first saw its PIDs this session. This is the 'first-class culprit' attribution. Appears only for Claude/Codex with >0 written.
- **Data source:** public libproc proc_pid_rusage(pid, RUSAGE_INFO_V4).ri_diskio_byteswritten (cumulative block bytes since process start) with ri_proc_start_abstime for PID-reuse detection; WorkloadSampler keeps [Int32:(written,startAbs)] baselines and a [String:UInt64] per-label session accumulator, clamping negative deltas and re-baselining when startAbs changes.
- **Why simplest:** residentMB(for:) ALREADY calls proc_pid_rusage(pid, RUSAGE_INFO_V4, &info) for every PID; reading info.ri_diskio_byteswritten and info.ri_proc_start_abstime from the same struct is zero extra syscalls. WorkloadSampler already groups all Claude Code / Codex PIDs under one label via add() (lines 148, 155), so summing per-label deltas across PIDs is already the right architecture. Same-user-only is not a practical limit (both tools run as the user).

## New Snapshot fields

- `diskWriteBytesPerSec: Double = 0` — System-wide host bytes/sec written to the SSD this tick (sum of IOBlockStorageDriver 'Bytes (Write)' deltas / elapsed). EXACT, no flag. Host bytes submitted to the block layer, not post-write-amplification NAND bytes. Smoothed for the rate label.
- `diskReadBytesPerSec: Double = 0` — System-wide host bytes/sec read ('Bytes (Read)' delta). EXACT, no flag. Shown for context alongside writes.
- `diskSessionWrittenGB: Double = 0` — Host GB written accumulated since app launch / since reboot (sum of positive Bytes(Write) deltas while primed). The IOKit counter resets on reboot, so this is explicitly 'since reboot'.
- `diskLifetimeWrittenGB: Double = 0` — Cumulative host GB written persisted across reboots = baseline (read from the JSONL log tail at startup) + this run's session bytes. Provenance is 'since Monitor Island install', NOT true drive lifetime (no SMART). The number that survives reboots and backs the damage estimate; a verifiable lower bound (misses time the app wasn't running).
- `diskWearPercent: Double = 0` — DERIVED perceived-damage estimate = diskLifetimeWrittenGB / (diskTBWAssumed*1000) * 100. BEST-EFFORT — must always render with '~' and 'est'. Not a hardware wear number.
- `diskWearBestEffort: Bool = true` — Always true in v1 (no SMART ground truth shipped). Mirrors tempBestEffort. Drives the '~'/'est' honest labeling. A future verified SMART PERCENTAGE_USED path would set this false.
- `diskTBWAssumed: Double = 1000` — Assumed Total-Bytes-Written endurance (TB) used as the wear denominator, selected by capacity tier (256GB->300,512GB->600,1TB->1000,2TB->2000). Apple publishes NO TBW; these are conservative community estimates surfaced in diskWearNote. Kept as a Snapshot field so --dump shows the assumption.
- `diskWearNote: String? = nil` — Honesty string, exact sibling of aneEstimateNote (Snapshot.swift:66): 'Apple does not publish TBW; estimate = host bytes / ~N TB rated; actual NAND wear is higher (write amplification).'
- `WorkloadEntry.diskWrittenSessionMB: Double = 0` — Per-culprit attribution: cumulative ri_diskio_byteswritten (block bytes) summed across all PIDs of this label since Monitor Island first saw them, in MB. Populated for Claude Code / Codex. 'session' scope (resets on app restart, re-baselines on PID reuse), block I/O, a lower bound on true NAND wear.
- `WorkloadInstance.diskWrittenSessionMB: Double = 0` — Per-PID MB written this session for the drill-down view; default 0 keeps Codable/back-compat.

## Persistence

- **storeLocation:** ~/Library/Application Support/MonitorIsland/disk_damage.jsonl — FileManager.urls(for:.applicationSupportDirectory,in:.userDomainMask).first + 'MonitorIsland' (isDirectory:true), created on first run via createDirectory(withIntermediateDirectories:true). The app is ad-hoc signed and NON-sandboxed (no sandbox entitlement — confirmed in Package.swift, only -undefined dynamic_lookup is added), so this path is unconditionally writable with no TCC prompt. Verified the dir does not exist yet, so first-run create is required.
- **format:** ONE append-only JSON-lines (.jsonl) plain-text file. Explicitly NOT SQLite (a WAL DB is exactly the Codex offender pattern). One newline-terminated JSON object per flush. The cross-reboot lifetime baseline is recovered by tailing the last line at startup (no second file).
- **frequency:** At MOST once every 300 s (5 min, hard-coded flushInterval with precondition(flushInterval>=60) so it cannot be misconfigured short) AND once on clean quit via AppDelegate applicationWillTerminate (add this method — it does not exist yet). In-memory accumulation between flushes; the 60Hz Smoother and the 1/2/5 s Sampler tick NEVER write to disk. Because terminate isn't guaranteed on force-quit/panic, the 5-min periodic flush bounds worst-case loss to 5 min.
- **schema:** {"ts":"<ISO8601>","sessionWrittenGB":<Double>,"lifetimeWrittenGB":<Double>,"damagePct":<Double>,"claudeCodeMB":<Double optional>,"codexMB":<Double optional>}. claudeCodeMB/codexMB omitted when 0 to keep lines lean. Example: {"ts":"2026-06-22T18:03:11Z","sessionWrittenGB":1.92,"lifetimeWrittenGB":4218.40,"damagePct":0.42,"claudeCodeMB":312.4,"codexMB":18.7}.
- **antiWearNotes:** Self-footprint is provably negligible: ~150 bytes/line x 288 lines/day ~= 43 KB/day ~= 16 MB/year ~= 0.0000026%/yr of an assumed 600 TBW (six orders of magnitude under the Codex bug's 37 TB/21 days). Enforced in CODE, not convention: precondition(flushInterval>=60) in DamageLogger.init; a single FileHandle.seekToEndOfFile()+write per flush (no per-tick writes, no fsync loop); JSONL not SQLite. Reboot reconciliation: lifetime = readLastLifetimeBytes() (JSONL tail) + thisRunSessionBytes; on first run / missing file lifetime seeds from this run only and is labeled 'since install'. Optional startup size guard: if the file exceeds a few MB, note it (multi-year safety).

## Honesty / accuracy labeling

- diskWriteBytesPerSec / diskReadBytesPerSec are EXACT (no flag) but are HOST bytes submitted to the block layer, NOT physical NAND writes — real NAND wear is higher by the controller write-amplification factor (typically 1.5-3x, unknowable from user space). Label 'host writes' / 'bytes to SSD', never 'NAND writes'.
- diskLifetimeWrittenGB provenance is 'since Monitor Island install', NOT a drive odometer. There is NO sudoless lifetime counter on Apple Silicon in v1 (NVMe SMART DATA_UNITS_WRITTEN is cut). The IOBlockStorageDriver counter resets every reboot; the persisted baseline accumulates only while the app has run. It is a verifiable lower bound. UI/log say 'since install'.
- diskWearPercent is DERIVED and best-effort: lifetime host GB / assumed TBW. diskWearBestEffort is hard-coded true in v1; the UI renders '~' and 'est'. Apple publishes NO TBW for any Apple Silicon SSD — diskTBWAssumed is a conservative capacity-tiered community estimate, surfaced in diskWearNote. Never show a bare percentage without the qualifier (mirrors the tempBestEffort / aneEstimateNote precedents already in Snapshot.swift).
- WorkloadEntry.diskWrittenSessionMB uses ri_diskio_byteswritten (the physical block-I/O field), NOT ri_logical_writes (which is ~2-39% higher, VFS-layer, and is not even present in rusage_info_v4). Using the logical field and calling it 'disk writes' would be a factual error. The number is 'block bytes written (session)', a lower bound on NAND wear, zero-floored from first PID sighting, re-baselined on PID reuse via ri_proc_start_abstime.
- Per-process attribution is same-user only (proc_pid_rusage returns -1/EPERM for other-user PIDs). Claude Code and Codex run as the user, so this is not a practical limit, but writes by any root helper they spawn are invisible — the figure is a lower bound, labeled 'session', not 'all'.
- System-wide IOBlockStorageDriver writes include EVERYTHING (OS, Spotlight, Time Machine, swap, the app itself), not just AI tools. The card must visually separate the system-wide Disk rate from the per-culprit Claude/Codex attribution so users don't read the system total as AI-caused.
- NVMe SMART 'Percentage Used' / 'Data Units Written' (the only WAF-aware, firmware-authoritative wear numbers) are deliberately ABSENT in v1. Do NOT fabricate a SMART-looking percentage from the derived estimate. A one-line TODO marks where a future verified SMART value could set diskWearBestEffort=false and drop the '~'.

## Risks & mitigations

- **NVMe SMART scope-creep: an implementer is tempted to add the 'real' wear % via the reverse-engineered IONVMeSMARTInterface vtable slot [5]. The ideation agents directly CONFLICT (some get SMARTReadData sudoless; others get kIOReturnNotPermitted / firmware 0x10000003 on the ANS3 controller), and the 'working' path calls a C plugin vtable that risks an arm64e PAC SIGABRT in a non-hardened binary. It adds the most code and the most fragility.** → Explicitly OUT of v1 scope. Ship the derived best-effort estimate (flagged). Leave a single-line TODO at the damage-computation site. If ever pursued, gate behind a fail-soft probe: attempt the open/read once, validate (PERCENTAGE_USED<=100, DATA_UNITS_WRITTEN plausible), and on ANY non-success or implausible value return nil and keep diskWearBestEffort=true. Never crash; never replace the estimate with a SMART-looking number unless firmware actually returned it. Prefer a CIOReport C shim over a Swift vtable call if attempted.
- **IOBlockStorageDriver counter resets to zero on reboot, so a naive implementation shows 0 lifetime/damage after every reboot — defeating the 'accumulate over time' goal.** → DiskSampler seeds its lifetime baseline from the JSONL log tail at init (DamageLogger.readLastLifetimeBytes) and reports lifetime = baseline + thisRunSession. The damage % uses lifetime, not the volatile session counter. The session row is explicitly labeled 'since reboot'. Verified by the manual reboot test.
- **PID reuse corrupts per-culprit attribution: a Claude/Codex PID exits and is recycled to an unrelated process before the next tick, so a cumulative-from-start counter appears to jump or drop.** → Baseline keyed on (ri_diskio_byteswritten, ri_proc_start_abstime); a changed start time signals reuse/restart -> reset baseline, contribute 0. Also clamp negative deltas to 0 (the same defense Network.swift uses for wrapping counters). Prune maps for PIDs absent from the current scan to bound memory.
- **snapToTargets()/--shot omission: if diskWrite/diskRead aren't added to Smoother.snapToTargets(), the --shot render and the first frame after wake show 0 for the disk rows even though the data is present.** → Explicitly listed in the Smoothing.swift edit: add diskWrite/diskRead to snapToTargets() alongside netDown/netUp. The --shot test step checks the rendered card shows non-zero disk rows.
- **Disk-image / external-drive entries inflate the system-wide rate; an external Thunderbolt/USB NVMe would also be a non-zero writer.** → Verified live that exactly one entry has non-zero Bytes(Write) and sum==that single internal-SSD total, so summing all entries is correct here (disk images report 0). 'Total host writes across attached block devices' is a defensible metric. If per-disk isolation is later needed, read kIOBSDNameKey ('disk0') or filter Physical Interconnect Location=='Internal'. The --sensors disk dump prints all entries so behavior is verifiable on any config.
- **Counter-jitter / wrap artifacts on the cumulative UInt64 deltas at 1-5 s poll (same class of issue Network.swift handles).** → DiskSampler is a literal structural copy of NetworkSampler: &- wrapping subtraction, max(0, rate) clamp, max(0.001, elapsed) — it inherits the battle-tested behavior.
- **Provenance confusion: users read 'SSD ... life' as the drive's true odometer (it is only 'since install') or read the system-wide Disk rate as AI-caused.** → Honesty labels are part of the deliverable: the wear row carries '~'+'est', diskWearNote spells out the TBW assumption and write-amplification caveat, the cumulative figure is 'since install', and the per-culprit sub-lines are visually separated under each tool. These mirror the shipped tempBestEffort/aneEstimateNote precedents.

## Decision rationale

The two candidate plans agree on ~90% of the design (both correctly mirror Network.swift for a delta sampler, both extend the existing proc_pid_rusage call in Workloads.swift at zero syscall cost, both use append-only JSONL flushed at most every 5 min, both cut NVMe SMART as fragile/conflicted, both flag wear best-effort exactly like tempBestEffort). I merged them and resolved the few real differences by the simplicity + robustness razor, verifying each load-bearing claim against the live machine and the SDK rather than trusting the ideation. WHAT I TOOK FROM PLAN 2 (the simpler persistence): a SINGLE disk_damage.jsonl whose last line is tail-read at startup to recover the cross-reboot lifetime baseline — strictly fewer files and fewer writes than Plan 1's separate lifetime_written.json that is rewritten every flush, so it is both simpler and better for the anti-self-wear constraint. Plan 2's ri_proc_start_abstime PID-reuse check is more precise than Plan 1's clamp-only, so I kept it (plus the clamp as belt-and-suspenders). Plan 2's capacity-tiered TBW (read the sudoless 'capacity' field, map 256/512/1024/2048GB -> 300/600/1000/2000 TB, default 1 TB) is more correct than Plan 1's hardcoded 600 and is cheap, so I adopted it — but cached the capacity read once (lazy let) so the tier is not recomputed per tick. WHAT I TOOK FROM PLAN 1: the explicit precondition(flushInterval>=60) as a CODE guard (not just convention) against the ironic self-wear failure, and the crisp 'sum all entries (==max on this machine)' verdict, which I confirmed live (one non-zero writer at 570985172992 bytes; five virtual entries at 0) — so summing needs no IORegistry parent-walking and is robust to a future external NVMe (it just becomes 'total host writes across attached block devices'). DELIBERATE CUTS: NVMe SMART is out of v1 because the ideation directly contradicts itself on sudoless feasibility on Apple Silicon (working via NVMeSMARTLib vtable for some agents; kIOReturnNotPermitted/0x10000003 on the ANS3 controller for others) and the 'working' path is a PAC-/vtable-slot-/OS-fragile private CFPlugin call for a number that updates ~daily and reads 0% on near-new drives — it fails the razor and is unnecessary to deliver all three required capabilities; a one-line TODO marks the future override point. I verified: IOBlockStorageDriver Statistics keys are exactly 'Bytes (Write)'/'Bytes (Read)' and readable sudoless (ran ioreg as uid 501); rusage_info_v4 contains ri_diskio_byteswritten (line 308) and ri_proc_start_abstime (line 299) and RUSAGE_INFO_V4=4 (so the existing residentMB call already fetches them); ri_logical_writes is NOT in v4 (starts v5), which both justifies using ri_diskio_byteswritten and avoids a struct-version change; the codebase uses the modern kIOMainPortDefault (GPU.swift) so Disk.swift matches; the app is non-sandboxed (Package.swift adds only -undefined dynamic_lookup) so ~/Library/Application Support is writable with no prompt; the Application Support dir does not exist yet (first-run create needed); and Codex's logs_2.sqlite + ~/.claude/projects exist as real attribution targets. UI stays minimal: expanded card only, all reuse of row()/Sparkline/fmtRate, zero new Theme colors (escalation reuses the already-allowed pressureWarn/pressureCritical), compact pill untouched per its documented 4-slot constraint.

## Considered but cut

- NVMe SMART ground truth (PERCENTAGE_USED / DATA_UNITS_WRITTEN via IONVMeSMARTInterface / NVMeSMARTLib): CUT from v1. Ideation agents directly conflict on sudoless feasibility on Apple Silicon (some get SMARTReadData working via the vtable; others get kIOReturnNotPermitted / firmware 0x10000003 on the ANS3 controller). The 'working' path calls a private CFPlugin vtable slot [5] that risks an arm64e PAC SIGABRT in a non-hardened binary, is vtable-slot- and OS-version-fragile, and returns a number that updates only ~daily and reads 0% on near-new drives. Unnecessary to deliver real-time + cumulative + attribution. One-line TODO left where a future verified value could override the estimate and clear diskWearBestEffort.
- Separate lifetime_written.json sidecar file (Plan 1): CUT in favor of tailing the single disk_damage.jsonl for the baseline — fewer files, fewer writes, better anti-self-wear.
- Summing-with-IORegistry-parent-walk to isolate the internal SSD: CUT. Live data shows disk-image/virtual entries already report Bytes(Write)=0, so summing all entries equals the single internal-SSD total with no parent-walking. The --sensors dump exposes per-entry stats if disambiguation is ever needed.
- Adding a 5th ring (SSD) to the expanded ring row / a 5th compact-pill slot: CUT. The pill's 4 slots are full by documented design and shrinking all rings to fit a 5th hurts legibility; a dedicated 'SSD~' detail row + WRITE sparkline conveys the damage at a glance without disturbing the familiar GPU/CPU/MEM/SWAP quartet.
- ri_logical_writes for per-process attribution: CUT (and it is not even in rusage_info_v4). ri_diskio_byteswritten is the correct block-layer wear field; logical writes would overstate and mislabel.
- UNUserNotificationCenter threshold alerting (Codex-bug early warning): CUT from v1 — needs NSUserNotificationUsageDescription added to the packaged Info.plist (package_app.sh). The visible WRITE sparkline + recoloring SSD~ row are the at-a-glance signal; alerting is a clean follow-up.
- User-configurable TBW override in SettingsPanel: deferred. v1 auto-selects a capacity-tiered default and surfaces the assumption in diskWearNote; a settings override is a small v1.1 follow-up.
- SQLite/any DB for the damage log: CUT as self-defeating — a WAL DB is exactly the Codex offender pattern. Append-only JSONL only.
